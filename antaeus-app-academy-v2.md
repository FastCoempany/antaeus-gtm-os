## What this document is

This is the current source-of-truth academy document for the product that now exists in the repo.

It is designed for:

- founders using the app for the first time
- first AEs or GTM operators inheriting the workspace
- internal operators explaining the product
- future customer success, support, or implementation work

It is not a developer deep-dive on every localStorage key. It is an operator manual for how the system works in practice.

## What Antaeus is

Antaeus is a founder-led sales operating system for the period before, during, and just after the first serious go-to-market hire.

It helps a team:

- define who to sell to
- research and prioritize accounts
- create live outbound motion
- run discovery with structure
- manage deal truth
- scope proof correctly
- deploy advisors intentionally
- pressure-test fragility
- calculate operating math
- judge readiness
- export a believable handoff kit

## What Antaeus is not

Antaeus is not:

- a CRM
- a generic sales training course
- a sequence sender
- a call recorder
- a forecasting platform
- a dashboard graveyard

The point is not to store everything.

The point is to create the operating truth a founder or first AE needs in order to stop running sales from memory.

## Current First-Session Truth

The current app does **not** start like the older academy described.

The real first-session flow is:

1. Sign up or sign in.
2. Complete onboarding.
3. Land in Welcome, not straight in Dashboard.
4. Use Welcome to understand the first ranked next move.
5. Optionally run the tour or open the demo lane.
6. Start creating operating truth in the first real module.
7. Return to Dashboard and Welcome as the system becomes more real.

This is important.

The app is now activation-first, not module-first.

## Part 1: The Journey

## Stage 1: Activate the workspace

### Goal

Turn a new account into a configured workspace with a believable first move.

### Surfaces used

- Sign up / Sign in
- Onboarding
- Welcome
- Dashboard
- Tour
- Demo lane

### What happens here

The user sets the core context that the rest of the app depends on:

- company name
- role
- stage
- product category
- quota
- ACV
- buyer context

That context is not supposed to die in onboarding. It should show up later in:

- Welcome
- Dashboard
- Quota Workback
- ICP Studio
- Discovery framing
- Settings

### What good looks like

By the end of Stage 1, the user should:

- understand what the app is for
- know what the first real move is
- see a workspace that reflects their context
- know they can come back to Welcome
- know the tour and demo lane exist

### Move on when

Move on when:

- onboarding is complete
- welcome is useful
- the top-ranked next action makes sense
- the user can get to a real module cleanly

## Stage 2: Define the market and the wedge

### Goal

Decide who is actually worth selling to.

### Surfaces used

- ICP Studio
- Signal Console

### What happens here

The user defines:

- industry
- company size
- geography
- buyer
- pain
- trigger
- proof window

Then they begin to test that wedge against live accounts and live signals.

### What good looks like

By the end of Stage 2, the user should have:

- at least one sharp ICP
- at least one live account in Signal Console
- at least one signal or account worth pursuing
- a clearer sense of what “broad” looks like versus “sharp”

### Move on when

Move on when:

- the ICP is specific enough to source against
- the first signal-backed account exists
- outbound no longer has to start from a blank guess

## Stage 3: Build the territory and account universe

### Goal

Turn target logic into a prioritized working universe.

### Surfaces used

- Territory Architect
- Sourcing Workbench
- Signal Console

### What happens here

The user converts market logic into:

- territory theses
- account tiers
- sourced prospects
- research-backed account priority

This is where the app should stop feeling like strategy and start feeling like operating context.

### What good looks like

By the end of Stage 3, the user should have:

- a thesis-backed territory
- accounts or prospects worth working
- account priority informed by signal, not just instinct

### Move on when

Move on when:

- the user can name who to contact next
- sourcing output is strong enough to support outbound
- Signal Console has context worth acting on

## Stage 4: Start motion across channels

### Goal

Create real contact motion instead of channel ideas.

### Surfaces used

- Outbound Studio
- LinkedIn Playbook
- Cold Call Studio

### What happens here

The user creates and logs:

- outbound angles
- touch strategy
- LinkedIn plays
- cold-call capture
- early motion evidence

This stage should create actual operating proof, not just copy drafts.

### What good looks like

By the end of Stage 4, the user should have:

- one usable outbound motion
- one logged touch or action
- one logged call or channel move
- enough motion truth for Dashboard to brief something real

### Move on when

Move on when:

- the system has at least one real motion artifact
- outreach is no longer hypothetical
- channel work is starting to compound into deals or calls

## Stage 5: Run the conversation with structure

### Goal

Turn meetings into intelligence, not just “good calls.”

### Surfaces used

- Call Planner
- Discovery Studio
- Deal Workspace

### What happens here

The user:

- prepares the call in Call Planner

- stores and advances deal truth in Deal Workspace

This is where the product should start feeling unmistakably operational.

### What good looks like

By the end of Stage 5, the user should have:

- one prepared agenda
- one discovery path with real logic
- one real deal with stage, next step, and pressure

### Move on when

Move on when:

- calls are no longer disconnected from deals
- discovery output survives the meeting
- a deal exists that can be reviewed, pressured, and advanced

Make the pipeline honest and the proof path explicit.

## Part 2: Surface Reference

## Activation and support surfaces

### Sign up

**Purpose:** create a real workspace owner account.

**Best used when:** the user is brand new.

**Primary input:** name, email, password, role.

**One-session output:** account creation and a clean path into onboarding or email confirmation.

**Feeds next:** Onboarding.

### Sign in

**Purpose:** return an existing user to the correct workspace state.

**Best used when:** the user already has an account.

**Primary input:** email and password.

**One-session output:** authenticated session routed into onboarding, welcome, or dashboard.

**Feeds next:** Onboarding, Welcome, or Dashboard.

### Onboarding

**Purpose:** initialize the workspace with enough context to make the rest of the system meaningful.

**Best used when:** the account is new or the user is resetting operating context.

**Primary input:** company, role, stage, category, quota, ACV, buyer context.

**One-session output:** activation context and seeded workspace configuration.

**Feeds next:** Welcome.

### Welcome

**Purpose:** explain the lay of the land and rank the next real move.

**Best used when:** the user has just finished onboarding or needs orientation again later.

**Primary input:** saved workspace state plus activation context.

**One-session output:** a believable first action and a clearer understanding of the system.

**Feeds next:** the top-ranked module, Dashboard, Tour, Demo lane, Methodology, or Settings.

### Dashboard

**Purpose:** act as the operating room for the workspace.

**Best used when:** the user wants to know what matters now.

**Primary input:** saved workspace summary across modules.

**One-session output:** next moves, pressure, coverage, and live operational truth.

**Feeds next:** almost every core module.

### Tour

**Purpose:** make the product understandable in one guided pass.

**Best used when:** the user wants the product story fast or the workspace is still sparse.

**Primary input:** current role plus saved workspace or demo state.

**One-session output:** a believable understanding of how the system compounds.

**Feeds next:** Dashboard and the most relevant adjacent modules.

### Demo lane

**Purpose:** load a believable sample workspace so the product never has to explain itself from empty state.

**Best used when:** the user wants to understand the system before building their own real workspace.

**Primary input:** scenario choice.

**One-session output:** seeded sample workspace in demo mode.

**Feeds next:** Dashboard, Tour, and all demo-critical modules.

### Settings

**Purpose:** give the user trust, backup, reset, and mode control.

**Best used when:** the user needs durability or recovery, or wants to understand what is real versus local.

**Primary input:** export, import, delete, category, demo-mode actions.

**One-session output:** more trustworthy workspace control.

**Feeds next:** backup recovery, demo transitions, product-category propagation.

## Targeting and intelligence surfaces

### Signal Console

**Purpose:** turn accounts and external signals into action-worthy research.

**Best used when:** the user needs account priority or account-specific context.

**Primary input:** account names, domains, research actions.

**One-session output:** researched accounts with heat, evidence, and next moves.

**Feeds next:** Outbound Studio, Call Planner, Dashboard.

### ICP Studio

**Purpose:** define who is actually worth selling to.

**Best used when:** the team needs a sharp wedge instead of generic “mid-market” thinking.

**Primary input:** industry, size, geography, buyer, pain, trigger, proof window.

**One-session output:** one or more scored ICP profiles with quality logic.

**Feeds next:** Territory Architect, Sourcing Workbench, Signal Console, Outbound Studio, Discovery framing.

### Territory Architect

**Purpose:** convert the wedge into a territory model.

**Best used when:** the team knows the ICP but needs a practical account-priority structure.

**Primary input:** theses, account candidates, tiering logic, territory choices.

**One-session output:** a more deliberate account universe.

**Feeds next:** Sourcing Workbench and Signal Console.

### Sourcing Workbench

**Purpose:** turn target logic into a working prospect universe.

**Best used when:** the team needs named accounts and contacts, not just category logic.

**Primary input:** sourcing criteria, personas, research effort.

**One-session output:** sourced prospects with quality reads and handoff potential.

**Feeds next:** Territory, Signal Console, Outbound.

## Channel execution surfaces

### Outbound Studio

**Purpose:** create the next outbound motion with enough context to actually send or log it.

**Best used when:** a real account, contact, or trigger exists.

**Primary input:** account, contact, persona, temperature, trigger.

**One-session output:** motion brief, quality read, and saved/logged angle.

**Feeds next:** Dashboard, Signal Console, LinkedIn Playbook, Handoff Kit.

### Cold Call Studio

**Purpose:** support a real cold call before, during, and after the call.

**Best used when:** the user is actively making calls or reviewing call outcomes.

**Primary input:** account choice, call state, objection, callback date, notes.

**One-session output:** call structure plus logged call truth.

**Feeds next:** Dashboard, Deal Workspace, Handoff Kit.

### LinkedIn Playbook

**Purpose:** make LinkedIn feel like a channel with real plays, not generic advice.

**Best used when:** the team wants account-specific credibility or follow-up motion.

**Primary input:** account context, persona, action type, recommended play.

**One-session output:** logged LinkedIn motion with clearer play context.

**Feeds next:** Dashboard, Handoff Kit.

## Call and deal execution surfaces

### Call Planner

**Purpose:** tighten the upcoming conversation before the meeting happens.

**Best used when:** the next important call is coming and needs a real agenda.

**Primary input:** contact, account, objectives, deal link, why-now context.

**One-session output:** discovery agenda with score, gates, and next move.

**Feeds next:** Discovery Studio and Deal Workspace.

### Discovery Studio

**Purpose:** run structured discovery live instead of improvising the call from memory.

**Best used when:** the user is in or immediately around a discovery conversation.

**Primary input:** discovery framework, meeting context, agenda handoff, call truth.

**One-session output:** worked moves, structured call logic, and downstream deal intelligence.

**Feeds next:** Deal Workspace, Readiness, Handoff Kit.

### Deal Workspace

**Purpose:** keep live pipeline truth inspectable, qualified, and pressure-tested.

**Best used when:** a real opportunity exists.

**Primary input:** deal stage, next steps, dates, qualification, notes.

**One-session output:** a deal with visible quality, pressure, and next-step logic.

**Feeds next:** Future Autopsy, PoC Framework, Advisor Deploy, Dashboard, Handoff Kit.

## Proof and risk surfaces

### Future Autopsy

**Purpose:** identify why a deal may die before it does.

**Best used when:** a deal is active and fragility needs to be made explicit.

**Primary input:** live deal context.

**One-session output:** failure themes and routed next actions.

**Feeds next:** Deal Workspace, Call Planner, Discovery Studio, PoC Framework.

### PoC Framework

**Purpose:** scope and run proof in a way that is measurable and transferable.

**Best used when:** a deal has moved into proof or pilot territory.

**Primary input:** linked deal, success criteria, boundaries, readout owner.

**One-session output:** saved proof plan with quality read and export.

**Feeds next:** Deal Workspace, Dashboard, Handoff Kit.

### Advisor Deploy

**Purpose:** deploy advisors on purpose, not as random credibility theater.

**Best used when:** a deal needs access, validation, social proof, or de-risking.

**Primary input:** advisor roster, linked deal, deployment moment, ask.

**One-session output:** logged advisor move tied to a real deal.

**Feeds next:** Deal Workspace, Dashboard, Handoff Kit.

## Planning and synthesis surfaces

### Quota Workback

**Purpose:** turn revenue ambition into operating math.

**Best used when:** the team needs to understand how much activity the motion requires.

**Primary input:** quota, ACV, conversion assumptions.

**One-session output:** touches, meetings, opps, and deals required.

**Feeds next:** Dashboard, Outbound Studio, Cold Call Studio, Readiness.

### Readiness Score

**Purpose:** tell the user whether the current motion is ready to scale or hire against.

**Best used when:** the founder needs an honest read on the operating system.

**Primary input:** cross-module workspace state.

**One-session output:** scored readiness with strongest areas, weakest areas, and biggest unlocks.

**Feeds next:** Welcome, Dashboard, Handoff Kit.

### Playbook

**Purpose:** assemble the operating narrative another human should inherit.

**Best used when:** the founder wants to see whether the system has become handoff-worthy.

**Primary input:** cross-module saved truth plus optional notes.

**One-session output:** a serious internal operating document and handoff export.

**Feeds next:** the first serious go-to-market hire.

## Part 3: System Rules

## Rule 1: Welcome comes before Dashboard for new users

The app should not throw a first-session user straight into a command surface and hope they infer the rest.

## Rule 2: Demo data must feel like the real product, not a side utility

The demo lane exists so the product can be believed in one session.

## Rule 3: Every core module should handshake into another module

No major surface should feel like a dead-end worksheet.

## Rule 4: The handoff kit is the supreme output

The system is not complete when the user has clicked every tab.

It is complete when another human could inherit the motion with much less founder memory.

## Rule 5: The app should not pretend to be a CRM

The point is operating truth, not record hoarding.

## Rule 6: Settings should increase trust, not fear

Backups, imports, reset, demo mode, and category settings should all feel legible and safe.

## Part 4: Two Fast Paths

## Fast Path A: Founder seeing the product for the first time

1. Finish onboarding.
2. Read Welcome.
3. Run the tour or demo if the workspace is sparse.
4. Create one ICP.
5. Research one account in Signal Console.
6. Create one deal.
7. Open Dashboard.
8. Open Playbook.

If those 8 steps feel believable, the product story is working.

## Fast Path B: Founder preparing for the first serious sales hire

1. Check Readiness Score.
2. Review Dashboard.
3. Tighten ICP.
4. Review active deals in Deal Workspace.
5. Run Future Autopsy on the most fragile deal.
6. Review PoC state if proof is in play.
7. Review Advisor Deploy if external leverage exists.
8. Open Playbook.
9. Export the handoff.
10. Export a backup from Settings.

If those 10 steps feel coherent, the product is doing its highest-value job.
