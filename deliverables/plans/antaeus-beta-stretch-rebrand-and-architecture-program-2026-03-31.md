# Antaeus Beta-Stretch Rebrand And Architecture Program

Date: 2026-03-31

Status: strategic control program for the final stretch before beta

Purpose: define the most ambitious pre-beta program yet, combining:

- complete rebrand direction
- complete module-architecture restructuring logic
- user-psychology and behavioral design rules
- market-pattern research
- implementation sequence
- founder-only decisions vs Codex-executable work

This document is not a light design memo. It is the operating program for taking Antaeus from “modernized product” to “beta-ready product with a coherent brand, coherent module architecture, and coherent user behavior.”

---

## 1. Beta-Stretch Outcome

The beta stretch should produce a product that feels like:

- one strategic operating system
- one opinionated brand
- one ranked decision environment
- one believable founder-to-operator handoff machine

It should **not** feel like:

- a CRM
- a bundle of modules
- an AI wrapper
- a prettier spreadsheet
- a thought-leadership site glued to a product

The beta stretch is successful when:

1. the product’s visual identity is unmistakable
2. the product’s logic is unmistakable
3. every important surface has one dominant behavioral job
4. users stop wondering what to do next
5. the app feels more consequential than almost everything else in its category

## 2. Research Signal

The current market direction is clear.

### 2.1 What leading products are doing

**Linear**

- Linear leans into configurable density, grouping, ordering, filtering, and search instead of decorative emptiness.
- It lets teams decide what information appears, how it is grouped, and what is relevant to the current view.

Sources:

- https://linear.app/docs/display-options
- https://linear.app/docs/search
- https://linear.app/docs/filters

Implication for Antaeus:

- the app should not chase oversized empty-screen SaaS aesthetics
- it should be calm, dense, and ranked
- users should be able to see only what matters now without losing the sense that the full system exists

**Attio**

- Attio’s current positioning is explicitly AI-native and context-native.
- Its call intelligence is framed as native to the CRM, with conversation insight automatically connected to the right records instead of living in a separate tool.

Sources:

- https://attio.com/
- https://attio.com/blog/introducing-call-intelligence

Implication for Antaeus:

- intelligence should live inside the operating object, not as a sidecar gimmick
- module outputs should become native system truth immediately
- every conversation, signal, proof move, and handoff artifact should visibly compound

**Clay**

- Clay’s newer product language pushes centralized signals, guided setup, and actionable workflows instead of isolated enrichments.
- The Signals hub is positioned as a single place to track and manage signals with step-by-step setup.

Source:

- https://www.clay.com/changelog/2025-04-18

Implication for Antaeus:

- signals should not feel like raw research inventory
- the user should always know which account matters, why it matters, and what action should follow

### 2.2 UX psychology that should govern the beta stretch

**Progressive disclosure**

- NN/g: progressive disclosure defers advanced or rarely used features to a secondary screen, making applications easier to learn and less error-prone.

Source:

- https://www.nngroup.com/articles/progressive-disclosure/

Implication for Antaeus:

- the first screen on every module should show the primary move, not every possible move
- advanced settings, secondary actions, and long-tail controls should appear only when they become relevant

**Recognition over recall**

- NN/g: make information and interface functions visible and easily accessible
- contextual tips beat generic tutorials because users do not retain big abstract walkthroughs

Source:

- https://www.nngroup.com/articles/recognition-and-recall/

Implication for Antaeus:

- do not make users remember the system
- show current object, current pressure, current next move, and current downstream consequence
- prefer contextual prompts over generic education

**Information scent**

- NN/g: users decide where to go based on how likely the source is to answer their question and how long it will take to get the answer

Source:

- https://www.nngroup.com/articles/information-scent/

Implication for Antaeus:

- labels, CTA copy, nav names, and cross-module handoffs must be concrete and consequence-driven
- “Explore,” “Learn more,” “Bridge,” and “System” language should be used carefully
- every route should tell the user what they will get, not just where they are going

## 3. Core Product Thesis For Beta

Antaeus is a founder-to-first-operator revenue operating system.

Its job is not to store customer records.

Its job is to:

- turn fuzzy founder motion into visible operating truth
- surface the next commercially important action
- pressure-test deals before they rot
- turn proof, signals, discovery, and handoff into one compounding system

So the beta stretch should optimize for:

- decision quality
- pressure visibility
- handoff readiness
- speed to next meaningful move

Not for:

- exhaustive record management
- generic collaboration
- infinite configurability
- feature parity with CRMs

“Complete rebrand” should be treated as 5 layers.

### 4.1 Brand narrative

Define:

- what Antaeus is
- what it is not
- why it exists
- who it is for
- what emotional territory it should own

Current strongest territory:

- severe clarity
- operating truth
- founder honesty
- handoff readiness
- pressure over vanity

### 4.2 Verbal identity

Lock:

- voice principles
- naming rules
- CTA rules
- state language
- error language
- proof language

Recommended verbal direction:

- more specific
- more consequential
- less SaaS-polite
- less “AI assistant”
- more like an operating memo than a marketing page

### 4.3 Visual identity

Lock:

- typography system
- motion rules
- state colors
- density rules
- icon rules
- public-perimeter vs app-interior relationship

### 4.4 Interaction identity

Lock:

- how next moves are expressed
- how risk is expressed
- how quality is expressed
- how handoffs are expressed
- how “seriousness” is expressed

### 4.5 Brand perimeter

Unify:

- coming soon
- methodology
- demo lane
- purchase
- auth corridor
- app shell

So the brand does not fracture at the boundary.

## 5. Module Architecture Thesis

Every module should stop behaving like a page and start behaving like an operating board.

Each board should answer 5 questions:

1. What object am I working on?
2. What is the pressure on that object right now?
3. What is the single best move now?
4. What changes downstream if I do it?
5. What should the system remember automatically after I act?

This is the architecture rule that should govern the rest of the beta stretch.

### 6.1 Activation family

Surfaces:

- auth
- onboarding
- welcome
- dashboard
- demo lane

Job:

- get the user from curiosity to belief to first real action with minimal friction

Restructure rule:

- reduce orientation content
- increase contextual activation
- keep one dominant next move on every surface

### 6.2 Targeting family

Surfaces:

- ICP Studio
- Territory Architect
- Sourcing Workbench
- Signal Console

Job:

- translate market ambiguity into a ranked working list

Restructure rule:

- upstream truth should always tighten downstream work
- no target surface should feel like disconnected data entry

### 6.3 Motion family

Surfaces:

- Outbound Studio
- Cold Call Studio
- LinkedIn Playbook

Job:

- turn account truth into contact motion

Restructure rule:

- channels should feel like variants of one motion system
- every motion should inherit the same account, signal, and priority truth

### 6.4 Execution family

Surfaces:

- Call Planner
- Discovery Studio
- Deal Workspace
- Future Autopsy
- PoC Framework
- Advisor Deploy

Job:

- move active commercial work forward without losing pressure truth

Restructure rule:

- these surfaces should feel consequential, not educational
- diagnosis must always turn into corrective action

### 6.5 Synthesis family

Surfaces:

- Quota Workback
- Readiness
- Founding GTM / Handoff Kit
- Settings

Job:

- convert all prior system truth into strategic clarity, trust, and transfer readiness

Restructure rule:

- summary surfaces should not repeat module noise
- they should rank pressure, expose maturity, and show what still blocks a safe handoff

## 7. Behavioral Laws

These should govern the beta stretch more strictly than visual preference.

### Law 1: One dominant move

Each important screen gets:

- one primary move
- one fallback move
- everything else quieter

### Law 2: Context before controls

Show:

- object
- pressure
- quality
- next move

Before showing a wide field of controls.

### Law 3: Do not ask twice

If the system already knows:

- product category
- role lens
- ICP quality
- deal stage
- proof pressure

Then downstream modules should inherit that truth instead of asking again.

### Law 4: Diagnosis must route into action

No diagnostic surface should end as “interesting insight.”

Every such surface must route into:

- the corrective module
- the corrective object
- the corrective next move

### Law 5: Intelligence must become native system truth

Signals, call notes, proof state, risk reads, and readiness reads should not feel like detached analysis artifacts.

They should immediately feed:

- the object header
- the next action queue
- the handoff state

### Law 6: Severe honesty wins

Antaeus should be allowed to say:

- this target is too broad
- this deal is weaker than the stage says
- this proof is thin
- this thread has no champion
- this motion is not handoff-safe

That honesty is part of the brand.

## 8. Beta-Stretch Program Phases

### Phase A: Rebrand truth lock

Goal:

- decide the final brand position and verbal identity

Outputs:

- brand positioning memo
- voice rules
- CTA rules
- naming rules
- state language rules

Codex can do:

- write the memo
- build the voice system
- audit current copy against it

Founder input needed:

- final brand severity level
- whether “Antaeus” naming stays unchanged
- whether the public voice should remain severe vs softened for broader marketability

### Phase B: Information architecture reset

Goal:

- re-map the full app around object flow, not module adjacency

Outputs:

- object map
- nav map
- cross-module path map
- “least path of resistance” route map

Codex can do:

- audit the current architecture
- propose the new map
- patch the shell and module routes

Founder input needed:

- none for first-pass design

### Phase C: Module operating-board rewrite spec

Goal:

- define the top-board structure for every module family

Outputs:

- family-by-family operating-board specs
- decision hierarchy rules
- quality/risk/next-move standards

Codex can do:

- write and implement the specs

Founder input needed:

- only if you want to override the opinionated structure

### Phase D: Rebrand execution across the perimeter

Goal:

- make the public edge feel like the same product as the interior

Outputs:

- final coming soon
- methodology brand alignment
- demo lane brand alignment
- purchase corridor brand alignment

Codex can do:

- design and implement the perimeter surfaces

Founder input needed:

- only if you want to change public commercial tone or conceal/reveal more

### Phase E: Architecture convergence by family

Goal:

- remove the last “module island” feeling

Outputs:

- shared object headers
- shared next-move logic
- shared pressure language
- shared handoff language

Codex can do:

- implement this in code

Founder input needed:

- none for first pass

### Phase F: Beta research operating system

Goal:

- make beta learning deliberate instead of anecdotal

Outputs:

- beta tester rubric
- session script
- observation sheet
- severity rubric
- decision log

Codex can do:

- create all of these artifacts

Founder input needed:

- who the first beta users are
- how many
- whether the beta is free, paid, or design-partner based

### Phase G: Final beta gate

Goal:

- decide whether the product is truly ready for guided beta

Outputs:

- beta-readiness gate
- remaining blocker list
- launch-mode recommendation

Codex can do:

- run the audit and produce the gate

Founder input needed:

- final commercial stance
- final risk tolerance

## 9. What Should Be Rebranded Versus What Should Stay

### Should be rebranded hard

- public narrative
- public typography hierarchy
- CTA language
- module family naming tone
- state labels
- commercial corridor language

### Should be restructured hard

- object inheritance
- next-move ranking
- cross-module handoff clarity
- diagnostic-to-action routing
- proof and handoff visibility

### Should stay stable

- the core premise that this is not a CRM
- the module families themselves
- the emphasis on handoff, pressure, proof, and operating truth
- the dark, serious, command-surface character of the app interior
