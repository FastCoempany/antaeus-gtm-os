# Antaeus Architecture Truth Memo

Date: 2026-04-01

Status: canonical architecture truth for the final stretch before beta

Purpose: lock where Antaeus is going so architecture, redesign, rebrand, prototype work, and implementation sequencing stop drifting.

This memo is not a speculative brainstorm.
It is the governing synthesis of:

- the current product truth
- the Academy material
- the rebrand and IA reset work already completed
- the architecture research brief
- the behavioral psychology framing
- the clarified founder position from the Claude thread

---

## 1. Executive Truth

Antaeus is not trying to become a prettier multi-module sales app.

Antaeus is trying to become a command-first, object-based go-to-market operating system that:

- ranks commercial pressure
- carries context forward automatically
- preserves deep strategic work
- transforms open loops into downstream open loops
- feels visually like a premium modern instrument, not a file cabinet

The current app's main weakness is not lack of depth.
It is the hallway.

Today the user still experiences too many equal-weight doors.
The architecture reset exists to replace that hallway with a smarter front door.

---

## 2. The Core Architecture Shift

### Current architecture problem

The current app is still largely experienced as:

- module first
- page first
- navigation decision first

That means the user repeatedly answers:

- where do I go now?
- which module matters?
- where does this context live?

Even when the modules themselves are strong.

### Target architecture

The target architecture is:

- command first
- object first
- lens second
- handoff always visible

The new logic is:

1. rank the object under pressure
2. inspect it quickly
3. go into the correct deep-work lens
4. carry truth forward into the next loop

So the new system is not:

- module -> page -> local controls

It is:

- command -> object -> lens -> downstream loop

## 4. What Must Change

The following must change before beta if Antaeus is going to feel like the product it wants to be.

### 4.1 Navigation logic

The user must stop starting from:

- a flat list of destinations

The user must start from:

- ranked work
- ranked objects
- visible tension
- one dominant next move

### 4.2 Command center of gravity

Welcome and Dashboard as separate mental models must dissolve into a single command surface with multiple densities.

That command surface is:

- the real welcome
- the real dashboard
- the real daily operating center

### 4.3 Context continuity

The app must stop forcing the user to re-explain the same object in multiple modules.

Objects should carry:

- state
- pressure
- gaps
- connections
- next move
- downstream effects

### 4.4 Surface text philosophy

The app is too text-heavy at the surface today.

That does not mean the product should become shallow.

It means:

- more spatial hierarchy
- more metric-first communication
- less explanatory burden at the top layer
- more detail revealed at the correct depth

### 4.5 Visual identity

The old visual language is not the future.

The architecture reset must be born inside the new identity:

- brighter
- cleaner
- more modern
- calmer
- less dark/warm-heavy
- less generic B2B admin UI

The product should feel:

- inviting
- tempting
- expensive
- structurally intelligent

## 5. The Four Layers

The architecture reset is governed by four interaction layers.

These are not four pages.
They are four modes of interaction.

### Layer 1: Command

Job:

- answer what matters now
- rank the work
- explain why
- show the next move

This layer contains the three densities of the same system truth:

- Brief
- Grid
- Queue

### Layer 2: Sheet

Job:

- inspect the object
- understand the context quickly
- see the gaps
- take a light action
- decide whether to go deeper

The sheet must be denser than the card or brief block that opened it.

It is not allowed to feel like a blurrier version of the card.

### Layer 3: Workspace

Job:

- do the actual work
- preserve flow
- keep object continuity visible
- let the deep frameworks remain fully intact

This is where the existing module depth continues to live.

### Layer 4: Graph

Job:

- show the user the system they built
- reveal compounding they did not consciously notice
- act as rare reward and diagnostic view

This is not the main navigation paradigm.

It is hidden, rewarding, and deliberately non-primary.

## 6. The Behavioral System

Antaeus is not just using behavioral psychology as decoration.
It is being architected around it.

### 6.1 Zeigarnik effect

The app should create productive tension through visible incompleteness.

The key rule:

open loops should not disappear.
They should migrate forward.

That means:

- completing work should transform the object's state
- it should expose the next downstream loop
- the user should stay inside a living commercial story

Wrong:

- item done -> item gone

Right:

- item done -> next consequence surfaced

### 6.2 Variable reinforcement

The product should sometimes surprise the user with meaningful compounding they did not explicitly ask to see.

Examples:

- a signal cluster becoming unusually strong
- a proof plan raising readiness more than expected
- the hidden graph showing unexpected operating density

This is not gamification.
It is the system noticing the user's work.

### 6.3 Endowed progress

The user should feel in motion quickly.

That means:

- onboarding seeds real command items immediately
- the command surface should never feel blank and punishing
- the workspace should visibly densify as truth accumulates

### 6.4 Loss aversion

The user's effort must become visibly embodied in the system.

That means:

- work should compound across surfaces
- context should appear in the next place automatically
- abandoning the app should feel like losing a living operating structure

### 6.5 Cognitive load reduction

The app should reduce:

- typing
- context restatement
- module choice
- uncertainty about what matters now

The app should increase:

- obvious defaults
- ranked next actions
- one-click continuation
- visible consequences

### 6.6 Sensory anchoring

The interface should force attention structurally.

The oversized metric is not decorative.
It is a behavioral forcing function.

The user should feel the state before reading the explanation.

Examples:

- 11 days
- 62 heat
- 84 quality

That pre-cognitive hit is part of the architecture, not just part of the styling.

## 7. State Language Lock

Abstract ranking language is not enough.

The command system should prefer state language over generic priority language.

Preferred:

- Urgent
- Active
- Healthy

Avoid as the primary visible language:

- High
- Medium
- Low

Reason:

- state labels describe the condition of the object
- they imply direction
- they create more specific emotional response

And the `⊘` gap indicator is now a structural device, not a decorative one.

It should remain visible across layers wherever incompleteness matters.

## 8. Visual Identity Truth

The old product look is not the target.

The architecture reset must not inherit the visual feeling of the old app shell.

### What the new identity should feel like

- bright
- modern
- high-conviction
- airy
- calm density
- premium

### What it should not feel like

- generic dark SaaS
- blue-black CRM admin UI
- warm-dark builder aesthetic
- empty prettiness
- AI glowware

### Visual salience rule

Brightness and contrast should be rationed.

They should be used for:

- the single next move
- real risk
- confidence truth
- save/unsaved truth
- the primary metric anchor

Everything else should stay quieter.
