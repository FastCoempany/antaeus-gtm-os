## 1. Executive Position

Antaeus should move from a module-first app toward an object-centric operating system.

But beta should **not** be the moment when the product abruptly collapses every module into a brand-new navigation model.

So this reset program takes the research brief’s destination and imposes a safer implementation rule:

- preserve familiar entry points
- change the behavior underneath them
- make objects, ranked work, and shared context increasingly dominant
- defer hard structural collapses until the product has survived beta evidence

That is the core posture of this program.

---

## 2. What This Program Is Solving

The current product is much stronger than it was.

But it still carries the legacy of being built as:

- many modules
- many entry points
- many local decisions

The IA reset exists to solve 5 structural problems:

1. users still think in terms of pages more than objects
2. ranked work exists, but not yet as the product’s universal logic
3. context carries forward better than before, but still not fully enough
4. some modules still feel like destinations instead of lenses on shared truth
5. the app still asks the user to navigate more than a mature operating system should

---

## 3. Non-Negotiable Constraints

These constraints come from the rebrand and research work and are now locked for the IA reset.

### Constraint 1

Antaeus is not a CRM.

The IA must not evolve toward:

- record ownership maze
- admin-first architecture
- tab proliferation
- database-as-UI

### Constraint 2

The IA must reduce cognitive overhead, not just rearrange it.

If a change creates more navigation or more uncertainty before it creates more clarity, it is the wrong change.

### Constraint 3

The IA must preserve commercial seriousness.

The product should feel like:

- ranked pressure
- strategic clarity
- operating consequence

Not:

- workflow administration
- productivity gamification
- AI theater

### Constraint 4

Beta must not be used as an excuse for architectural recklessness.

If an IA change would:

- break learned navigation too abruptly
- explode QA burden
- invalidate current docs and demo motion overnight

it should be staged.

## 4. Target IA Model

The target model is a four-layer interaction system:

1. Command
2. Sheet
3. Workspace
4. Graph

This is not a four-screen template.
It is a four-mode interaction model.

### 4.1 Command

Job:

- rank what matters now
- summarize what changed
- reduce the “what do I do next?” burden

Primary modes:

- Brief
- Grid
- Queue

This is where:

- urgency
- decay
- gap severity
- value magnitude
- downstream impact

should become visible.

### 4.2 Sheet

Job:

- let the user inspect an object quickly without fully changing context

This is the “open the thing without losing the larger ranked field” surface.

It should show:

- current object truth
- current pressure
- missing gaps
- recommended next move
- downstream consequences

### 4.3 Workspace

Job:

- let the user do real work on the active object

This is where modules increasingly become:

- views
- tabs
- operating boards

on top of shared objects.

### 4.4 Graph

Job:

- show the system-level web of invested work and compounding truth

For beta, Graph should remain:

- diagnostic
- celebratory in rare moments
- optional

It should not become a daily dependency.

## 5. Object Model

The object model should become the real architecture of the app.

### Primary operating objects

- ICP
- Account
- Signal
- Motion
- Call
- Deal
- Proof
- Advisor
- Handoff

### System objects

- Readiness
- Quota plan
- Activation context
- Workspace trust state

### Core IA rule

Every important surface should answer:

1. what object is this
2. what pressure is on it
3. what is the best next move
4. what changes downstream if I act
5. what will the system remember automatically

If a surface cannot answer those five questions, it is architecturally weak.

## 6. Module Family Mapping

This is the beta-safe restructuring map.

### 6.1 Command family

Current sources:

- Dashboard
- Signal Console
- Readiness
- week-one lifecycle
- welcome prioritization

Reset direction:

- Dashboard becomes the home of ranked operating truth
- Signal Console feeds the command layer as the signal engine
- Readiness feeds quality and pressure scoring
- Welcome remains an activation command layer for new users

### 6.2 ICP / Account family

Current modules:

- ICP Studio
- Territory Architect
- Sourcing Workbench

Reset direction:

- ICP becomes a shared scoring/config object
- Territory becomes the account operating lens
- Sourcing becomes the account-add / account-qualify lens

Beta-safe rule:

- preserve the current modules as entry points
- but make them increasingly feel like lenses over the same account graph

### 6.3 Signal / Motion family

Current modules:

- Signal Console
- Outbound Studio
- LinkedIn Playbook
- Cold Call Studio

Reset direction:

- Signal creates priority
- Motion becomes the unifying object
- channels become channel lenses on the same motion truth

Beta-safe rule:

- keep the current channel modules
- unify the object headers, context, and follow-through logic underneath them

### 6.4 Call / Deal / Proof family

Current modules:

- Call Planner
- Discovery Studio
- Deal Workspace
- Future Autopsy
- PoC Framework
- Advisor Deploy

Reset direction:

- Deal becomes the dominant execution object
- Call, Discovery, Risk, Proof, and Team become deal-adjacent working lenses

Beta-safe rule:

- preserve each major surface for now
- but increasingly make them feel like connected deal boards, not disconnected destinations

### 6.5 Synthesis / transfer family

Current modules:

- Quota Workback
- Readiness
- Founding GTM / Handoff Kit
- Settings

Reset direction:

- these surfaces become the strategic compression layer
- they should show verdicts, maturity, transfer readiness, and trust boundaries

## 7. Navigation Reset

### 7.1 Current problem

The nav still largely reflects historical module structure.

That is understandable, but no longer ideal.

### 7.2 Target state

Primary navigation should eventually read like:

- Command
- Targets
- Motion
- Execution
- Transfer

But beta should not jump there all at once.

### 7.3 Beta-safe navigation strategy

For beta:

- keep the current left-rail categories broadly recognizable
- strengthen category logic
- reduce cross-category confusion
- increase command-first routing

The navigation should become:

- less “choose a module”
- more “enter the object family / operating family”

### 7.4 Command palette role

The command palette should be built as:

- a universal accelerator
- object search
- action execution
- mode switching

But not as:

- the only sensible way to navigate

The mouse path must still be excellent.

---

## 8. Behavioral Rules For The Reset

### Rule 1: Ranked work beats flat work

Every important family should expose:

- what matters first
- what is decaying
- what can wait

### Rule 2: Objects beat pages

A user should increasingly feel:

- “I am inside this deal”
- “I am inside this account”

not:

- “I am on a module page”

### Rule 3: Cross-object compounding must be visible

When a user acts, the system should show:

- where that truth now lives
- what it updated
- what that unlocks next

### Rule 4: Incompleteness must be framed as leverage, not shame

The app should say:

- “this missing champion weakens the thread”
- “documenting proof here sharpens the deal”

Not:

- “you are incomplete”

### Rule 5: Session arcs matter

Every session should:

- open with ranked clarity
- support fast action
- end with a visible next return trigger

## 10. What Changes Before Beta Versus After Beta

### Before beta

- command layer becomes more explicit
- object headers become stronger
- family convergence becomes more visible
- current module entries remain
- command palette becomes an accelerator

### After beta, if validated

- deeper nav simplification
- harder shift from module labels to family/object labels
- more aggressive consolidation of working lenses
- stronger graph/system-level views

This distinction is important.

The research brief points toward the destination.
Beta should validate the migration.

---

## 11. Acceptance Criteria

The IA reset is successful when:

1. new users can tell what object they are acting on
2. users no longer hesitate between multiple modules for the same next move
3. ranked work is visible on entry
4. cross-object compounding is obvious after action
5. family boundaries make more sense than the historical module map
6. the app feels more like an operating system and less like a module library

---

## 12. What Codex Can Execute

Codex can execute:

- command-layer redesign
- family convergence specs

- command palette spec and implementation

- whether beta users can tolerate more aggressive nav changes
- whether the public story should expose the new IA directly
- whether the product should rename any user-facing module labels

## 13. Founder Decision Points

No founder input is required to count this IA reset program document as complete.

But the following decisions will matter later:

1. whether the current module labels stay during beta
2. whether command palette becomes visible in public demos as a hero behavior
3. whether the product introduces family-first navigation labels before or after beta evidence

## 14. Verdict

The architecture research brief is strong enough to define the destination.

This program is the correct way to apply it:

- keep the brief's object-centric and command-first truth
- reject a reckless one-pass module collapse
- restructure Antaeus through staged family convergence and command-layer dominance

This is now the actual information architecture reset program for Antaeus.
