# Antaeus Visual Composition Families

Date: 2026-04-03

Status: canonical visible-family spec for wireframing and taste testing

Purpose: define the small set of visual composition families Antaeus is allowed to use so the product stops being rebuilt page-by-page as isolated layout problems.

These families exist to support the full visible rebuild of the product.

These are not just functional categories.

They are visible families with:

- distinct posture
- distinct air
- distinct density
- distinct action hierarchy
- distinct disclosure behavior

The governing rule is:

**rewrite the face, not the mind**

These families are allowed to reorganize how surfaces look, breathe, reveal, and prioritize.

They are not allowed to flatten what the rooms actually know, do, or mean.

---

## 1. Family Rules

Every important product surface must map to one primary composition family.

A family defines:

- first-visible-zone structure
- hierarchy rhythm
- allowed plane count
- allowed copy burden
- color posture
- what must feel loud
- what must feel recessive

No page should invent its own family unless the governing docs are updated first.

---

## 2. Family One: Threshold

Examples:

- Welcome
- Onboarding

Feeling:

- invitational
- bright
- composed
- transitional
- confidence-building

Job:

- move the user from setup or orientation into real work

Visible laws:

- one commanding statement
- one dominant next move
- progress visible but not gamified
- support context quiet and secondary
- very low menu burden

Avoid:

- giant hero plus supporting card pile
- orientation copy avalanche
- recovery/help surfaces competing in the center

---

## 3. Family Two: Command Chamber

Examples:

- Dashboard

Feeling:

- ranked
- precise
- calm under pressure
- instrument-like

Job:

- show what matters now
- show why
- let the user go deeper without browsing

Visible laws:

- one focal object or ranked order
- command density is the star
- support/trust surfaces pushed outward
- room access secondary and summoned, not central
- explanation compressed into tight reasons

Avoid:

- dashboard vanity metrics as decoration
- equal-weight boxes
- room-browsing center stage

---

## 4. Family Three: Live Instrument

Examples:

- Signal Console
- Outbound Studio
- Cold Call Studio
- Call Planner

Feeling:

- live
- tense
- immediate
- operational

Job:

- support real execution in progress

Visible laws:

- top of page behaves like a working console, not a report
- action controls are real and proximal
- context visible but compressed
- health is a short pulse, not a summary essay

Avoid:

- tutorial-like intros
- too many stacked advisory cards
- making live rooms look like admin forms

---

## 5. Family Four: Decision Bench

Examples:

- ICP Studio
- Territory Architect
- PoC Framework

Feeling:

- deliberate
- sharpened
- exacting
- constructive

Job:

- help the user shape a strategic object with downstream consequence

Visible laws:

- the object being sharpened is visually central
- the top should show what quality or truth is being improved
- builder controls support the object, not the other way around
- proof of downstream consequence stays visible

Avoid:

- worksheet energy
- too many equal-weight builder panels
- “fill out the form” feeling

---

## 6. Family Five: Diagnosis Table

Examples:

- Deal Workspace
- Future Autopsy
- Discovery Studio

Feeling:

- severe
- investigative
- consequence-aware
- corrective

Job:

- show what is weak
- show what changed
- point to the corrective move

Visible laws:

- first-visible zone should expose pressure fast
- risk and truth are legible before long explanation
- corrective route is obvious
- health language should feel like operating truth, not analytics theater

Avoid:

- abstract diagnosis without corrective motion
- dramatic narrative copy at the top
- boxed warning-board clutter

---

## 7. Family Six: System Ledger

Examples:

- Readiness
- Quota Workback
- Playbook

Feeling:

- earned
- synthesizing
- steady
- authoritative

Job:

- summarize multi-room truth without becoming vague dashboard fluff

Visible laws:

- the system should feel like it is reconciling evidence
- one summary state should dominate
- secondary metrics must support that state
- compact system-health pattern is acceptable

Avoid:

- “BI dashboard” composition
- too many scoring widgets
- decorative metrics with no consequence

---

## 8. Family Seven: Trust Annex

Examples:

- Settings
- backup flows
- auth
- legal/privacy
- purchase

Feeling:

- calm
- trustworthy
- plainspoken

Job:

- keep the user safe
- make trust and recovery real

Visible laws:

- no drama
- no internal architecture language
- clear recovery moves
- clear trust signals

Avoid:

- overdesigned utility chrome
- fake product excitement

---

## 9. Mapping Rules

Each room should map as follows unless later updated:

- Welcome -> Threshold
- Onboarding -> Threshold
- Dashboard -> Command Chamber
- Signal Console -> Live Instrument
- Outbound Studio -> Live Instrument
- Cold Call Studio -> Live Instrument
- Call Planner -> Live Instrument
- ICP Studio -> Decision Bench
- Territory Architect -> Decision Bench
- PoC Framework -> Decision Bench
- Deal Workspace -> Diagnosis Table
- Future Autopsy -> Diagnosis Table
- Discovery Studio -> Diagnosis Table
- Readiness -> System Ledger
- Quota Workback -> System Ledger
- Playbook -> System Ledger
- Settings / auth / legal / purchase -> Trust Annex

---

## 10. Global Family Constraints

Across all families:

- no surface should exceed 3 dominant visual planes
- chips must be semantic, not decorative
- green must remain restrained
- orange must be rationed
- box count must stay low
- explanatory copy must defer downward whenever possible
- the shell must support, not dominate

---

## 11. What This Enables

These composition families let us:

- taste-test with discipline
- redesign multiple rooms without inventing a new visual language each time
- preserve room brains while changing the face of the product
- make Antaeus look authored instead of locally patched

They are a vehicle for rebuilding the visible language of the whole product without casually rewriting the strategic core.
