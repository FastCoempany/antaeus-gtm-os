# Antaeus Design Principle Strict Bible

This is the strict visual-and-semiotic source of truth for Antaeus.

Its purpose is not to collect every planning file in the repo.
Its purpose is to hold the files that govern how the app must **mean**, **feel**, and **signal** at every layer and every corner of the product.

This is the working cathedral for:
- semiotics
- symbol language
- visual identity
- facial architecture
- preservation guardrails
- canonical control artifacts

Original repo files remain in their source locations.
This bible is a **curated mirror and lock library** so the governing material can be reviewed in one place without breaking older links across the repo.

## Core laws

- Rewrite the face, not the mind.
- Reduce explanatory chrome, preserve strategic prose.
- Every sign must earn its meaning.
- Repetition creates comprehension.
- Decorative signals are forbidden.
- A meter, trace, chip, accent line, or icon cannot exist unless it maps to a real underlying system.
- If a sign implies a scale, the product must truly support that scale.
- If a mark means one thing in one room, it cannot mean something else somewhere else.
- The visible system must help the user understand pressure faster, not merely decorate pressure.
- Module brains, GTM frameworks, strategic logic, and room substance may be touched in presentation, but not changed in meaning.

## Immediate design warnings now locked

These are active constraints coming out of the current dashboard studies:

- No fake heat meters.
- No filler chips.
- No boxes-in-boxes-in-chips for the sake of occupancy.
- No wrapping signal chips that break the visual system because labels happen to be longer.
- No accent colors that accidentally fight the rest of the signal language.
- No component can imply seriousness it has not structurally earned.

## Naming language

For Antaeus, the preferred language is:

- **symbol lexicon** for the bank of branded signs and marks
- **semiotic system** for the meaning structure across the UI
- **design system** for the broader visual and component governance

Together, they form the visible operating language of the product.
