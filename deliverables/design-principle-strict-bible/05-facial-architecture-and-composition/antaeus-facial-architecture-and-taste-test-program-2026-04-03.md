# Antaeus Facial Architecture And Taste-Test Program

Date: 2026-04-03

Status: canonical visible-layer program for the next stage of the architecture reset

Purpose: define the visual-renovation program that sits on top of the completed architectural rewiring so Antaeus stops being structurally smarter than it looks.

This program governs the full visible rebuild of the product, not selective surface cleanup.

This program exists because the underlying operating logic, continuity, and command architecture are now materially in place, but the visible product expression still carries too much of the old interface DNA:

- card accumulation
- rail heaviness
- chip noise
- text burden
- mixed typography rhythm
- internal architecture language leaking into user-facing surfaces

This document is not a replacement for the broader restructure stack.

It is the visible-expression program that should now govern wireframing, taste testing, and visual convergence before further production UI rewiring.

## 2. Executive Position

The next work is not more abstract IA language.

The next work is not more infrastructure-first patching.

The next work is the visible convergence of Antaeus into the product the research, rebrand, and architecture work were actually aiming at:

- bright but severe
- calm under pressure
- visually expensive
- sparse where command should rule
- dense where inspection is earned
- unmistakably distinct from a CRM, admin shell, or dashboard bundle

This means the project is now entering a **facial architecture** phase.

That phrase means:

- how the product is composed
- how it carries air and pressure
- how it distributes authority
- how it uses type, color, container count, and disclosure
- how it stops looking assembled from legacy panels

This is not a reconsideration of the visible product language.

It is the execution of a full visible rebuild grounded in:

- the research
- the rebrand truth
- the behavioral psychology work
- the architecture truth
- the visual identity lock

---

## 3. Program Outcome

This program is successful when:

1. Antaeus no longer reads like a modernized version of the old shell.
2. Command, room entry, and deep rooms all feel authored inside the same visible language.
3. The shell no longer behaves or looks like a hallway of modules.
4. The app stops leaking architecture jargon into the user-facing surface.
5. The first-second impression feels premium, specific, and commercially serious.
6. Beauty and restraint reinforce the behavioral research instead of competing with it.
7. The product visibly reads as rebuilt, not patched.

This program is not successful if the app is merely:

- cleaner
- more collapsed
- more minimal
- more technically coherent

without also becoming:

- more beautiful
- more intentional
- more distinct
- more desirable

---

## 4. What This Program Locks

This program locks:

- the facial architecture rubric used to judge wireframes
- the visual composition families that replace page-by-page improvisation
- the canonical surfaces that must be taste-tested first
- the sequencing for moving from taste tests into production rewiring

This program does not lock:

- final component pixel values
- final production CSS tokens
- every room implementation detail

Those come later, after the canonical taste-test winners are selected.

---

## 4.1 Working Rule

The governing rule for this program is:

**rewrite the face, not the mind**

Meaning:

- the visible design language of the entire product is in scope for rebuild
- presentation structure may be rebuilt aggressively where needed
- module brains, GTM frameworks, strategic logic, and room substance may be touched in presentation

But:

- they may not be changed in meaning
- they may not be flattened for cleanliness
- they may not be weakened for beauty

## 5. Non-Negotiable Preservation Rules

The visible redesign may change:

- shell expression
- composition
- container usage
- copy burden
- disclosure timing
- rail behavior
- action hierarchy
- health surface presentation
- typography hierarchy
- container posture
- color emphasis
- disclosure sequencing

It may not dilute:

- GTM frameworks
- module substance
- strategic seriousness
- named premium assets
- cross-module compounding logic
- product thesis
- room-level intellectual depth

The hallway is allowed to disappear.

The rooms are not allowed to become shallow.

## 6. The Three Core Deliverables

This program produces three source-of-truth artifacts before more major production UI rewiring:

1. **Facial Architecture Rubric**
   - how any wireframe is judged
   - what fails automatically
   - what “beautiful Antaeus” means in practice

2. **Visual Composition Families**
   - the small set of visual room/surface families the product is allowed to use
   - the laws each family inherits

3. **Canonical Taste-Test Surface Matrix**
   - the first surfaces that must be wireframed and tested
   - what each one is proving
   - what must be preserved while it is redesigned

## 8. Program Principles

### 8.1 Beauty is not optional

Beauty is part of the product thesis because it affects:

- receptivity
- trust
- seriousness
- first-impression credibility
- felt inevitability

### 8.2 Restraint beats abundance

No screen should need beauty from:

- extra cards
- extra chips
- extra borders
- extra copy
- extra color

### 8.3 Surface identity must be earned

Each important surface should feel like a composed environment, not a page assembled from interchangeable dashboard parts.

### 8.4 Internal architecture language is not the product

The visible UI must not speak like:

- an implementation memo
- a systems-design note
- an internal architecture review

### 8.5 Visual calm cannot flatten consequence

The answer is not to make everything quieter until the app becomes generic.

The answer is to make the right things loud:

- pressure
- risk
- state truth
- one dominant next move
