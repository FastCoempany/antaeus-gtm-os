Date: 2026-04-02  
Status: Required preflight before `Phase 7 - Nav Re-architecture`

## 1. Purpose

This document exists to prevent Phase 7 from accidentally damaging the brain, soul, connective tissue, or strategic seriousness of Antaeus while the shell and navigation are re-architected.

Phase 7 is allowed to change:
- shell hierarchy
- access patterns
- room emphasis
- disclosure order
- copy burden
- default visibility
- command-first behavior

Phase 7 is not allowed to change, dilute, flatten, or casualize:
- GTM frameworks
- module substance
- methodology
- strategic seriousness
- product thesis
- cross-module compounding rules
- named premium assets
- the core logic inside `Signal Console`, `Future Autopsy`, `Discovery Studio`, `PoC Framework`, `Advisor Deploy`, and the rest of the deep rooms

This preflight is therefore not a copy-polish memo. It is a protection document and a behavioral architecture gate.

## 3. Why This Preflight Exists

Phase 7 is where the restructure becomes most dangerous.

Earlier phases let us:
- stabilize command surfaces
- stabilize the sheet
- preserve room-entry continuity
- align the room family
- demote the old nav without deleting it
- harden command intelligence

Phase 7 is different because it changes the shell itself. That means it is the exact phase where the following failures become likely if not explicitly forbidden:

- shell changes start rewriting room meaning
- copy cuts start deleting methodology instead of relocating it
- navigation simplification starts flattening strategic depth
- the desire for “clean” starts deleting useful strategic tension
- room intros are trimmed without protecting the logic they were carrying
- command-first access is achieved by weakening deep-room seriousness

This preflight exists so that does not happen.

## 4. Non-Negotiable Preservation Rules

The following are hard constraints.

### 4.1 Product Thesis Protection

Antaeus must remain:
- a GTM operating system
- a command-first operating room
- behaviorally engineered
- strategically serious
- deeper than market-default SaaS
- object-centric without becoming generic CRM software

Antaeus must not become:
- a polished filing cabinet
- a dashboard of widgets
- a generic AI assistant shell
- a shallow action list disconnected from GTM truth
- a soft motivational layer over weakened strategic rooms

### 4.2 Deep Room Protection

The following rooms remain preserved deep rooms with protected substance:

- `Signal Console`
- `Future Autopsy`
- `Discovery Studio`
- `Call Planner`
- `Deal Workspace`
- `PoC Framework`
- `Advisor Deploy`
- `ICP Studio`
- `Territory Architect`
- `Sourcing Workbench`
- `Outbound Studio`
- `Cold Call Studio`
- `LinkedIn Playbook`
- `Quota Workback`
- `Readiness`
- `Founding GTM / Handoff Kit`

Preserved deep room means:
- the room may gain better shell framing
- the room may gain better handoff context
- the room may gain better disclosure structure
- the room may not lose its strategic purpose
- the room may not be rewritten into a generic tool pane

### 4.3 Named Premium Asset Protection

The following are protected named assets and may not be genericized:

- `Signal Console`
- `Future Autopsy`

They must remain:
- named explicitly
- strategically legible
- severe and consequential where appropriate
- differentiated from generic tabs or commodity features

### 4.4 Connective Tissue Protection

Cross-module compounding is not auxiliary. It is part of the product’s intelligence layer and therefore protected.

The following continuity layer must remain intact:

- command ranking reflects real object pressure
- sheet explains why now and why this room
- room-entry bridge preserves context
- pinned context preserves object continuity
- return paths preserve originating mode
- saved actions compound into other rooms and shared state

### 4.5 Methodology Protection

The methodology may be:
- redistributed
- progressively disclosed
- compressed at the shell layer
- moved deeper into sheets, rooms, or methodology/help surfaces

The methodology may not be:
- deleted because it is “too much text”
- replaced with generic productivity language
- softened into vague encouragement
- rewritten in a way that weakens the strategic thesis

## 5. Behavioral Psychology Requirements

Phase 7 must express the behavioral spine already established by research.

### 5.1 Cognitive-Load Reduction

The shell must reduce menu burden and top-level decisions.

Required outcome:
- the user opens into command, not a hallway
- fewer equal-weight destinations compete
- room access becomes secondary and contextual

### 5.2 Zeigarnik / Ovsiankina Expression

The shell must keep unresolved work visible without turning the app into a wall of anxiety.

Required outcome:
- incomplete pressure remains visible
- open loops migrate forward
- the shell helps users resume the right work

Forbidden outcome:
- “everything complete” dead states
- loop closure that erases the next meaningful opportunity

### 5.3 Endowed Progress + Loss Aversion

The shell must preserve the sense that the user’s work compounds.

Required outcome:
- command surfaces know what the user already built
- rooms reflect carried objects and continuity
- the user sees that work has tentacles across the system

### 5.4 Least Resistance

The easiest path must be:
- open app
- see what matters
- inspect
- enter the right room
- return cleanly

Not:
- scan a large menu
- pick a room from memory
- reconstruct context

### 5.5 Structure Over Decoration

The shell must produce behavior through structure, not ornamental interface density.

Required outcome:
- hierarchy does the work
- disclosure does the work
- path control does the work

Forbidden outcome:
- excessive explanatory chrome
- panel soup
- decorative badges replacing real prioritization

## 6. Copy Burden Audit Thesis

The copy problem is not “too much text” in the abstract.

The copy problem is:
- important text placed too early
- methodology visible at the wrong layer
- the same truth repeated in command, rail, room intro, and footer
- paragraphs doing the job structure should do
- trust copy mixed with action copy
- strategic copy visible before the user has the context to use it

Therefore the governing rule is:

**Remove visibility burden before removing meaning.**

Or more operationally:

**Do not delete substance. Redistribute it to the correct layer.**

## 7. Disclosure Hierarchy by Layer

### 7.1 Command Layer

Command must answer only:
- what changed
- what matters now
- what should I do next

Command may include:
- one-line rationale
- ranked object reasons
- one dominant next move
- compact proof of system state

Command must not carry:
- long framework explanation
- methodology essays
- repeated room philosophy
- paragraphs that belong in the room or sheet

### 7.2 Sheet Layer

Sheet must answer:
- why now
- what is missing
- what compounds if I act
- why this room is the right continuation

Sheet may include:
- compact evidence
- compact gap logic
- continuity language
- one dominant room-entry CTA

Sheet must not become:
- a second dashboard
- a brochure
- a methodology article
- a place where four different educational blocks compete

### 7.3 Room First Fold

The first fold of a room must answer:
- what object am I on
- why did I come here
- what type of work is this room for
- what is the next serious move

The first fold may include:
- title
- compact subtitle
- bridge
- pinned context
- one composed setup zone

The first fold must not feel like:
- stacked cards explaining themselves
- a landing page for the room
- an essay about method before action begins

### 7.4 Room Deeper Canvas

The deeper room canvas may contain:
- strategic depth
- methodology
- frameworks
- diagnostic nuance
- operational detail

This is where substance belongs.

### 7.5 Methodology / Help Layer

This layer should hold:
- longer explanations
- rationale
- doctrine
- examples
- teaching

This content should be summoned, linked, or progressively disclosed.

It should not dominate the command surface or room first fold.

### 7.6 Trust / Utility / System Layer

This layer includes:
- sync state
- recovery state
- backup/export
- account/workspace notices
- error and resilience language

This copy must stay present enough to build trust, but not compete with command.

## 8. Copy Decision Framework

Every major text block in the app should be classified using the following decision rules.

### 8.1 Keep Visible

Keep visible if the text:
- determines the next move
- frames immediate strategic seriousness
- preserves object continuity
- prevents misuse or false confidence
- builds critical trust at the moment of risk

### 8.2 Compress

Compress if the text:
- is strategically right but too long
- repeats what the UI already shows
- can become one sentence or one line of labels
- is valuable but should not dominate the fold

### 8.3 Move to Sheet

Move to sheet if the text:
- explains why a selected object matters now
- explains missing information or pressure
- helps justify room entry
- is needed after object selection, not before

### 8.4 Move Deeper Into Room

Move deeper into room if the text:
- is methodology
- is process explanation
- is framework nuance
- is context the user needs only once they are already doing the work

### 8.5 Gate / Progressive Disclosure

Gate if the text:
- teaches rather than directs
- is not required for the immediate move
- is valuable but optional at the current moment
- serves users who want depth, not every user on every visit

### 8.6 Remove

Remove only if the text:
- is purely repetitive
- is decorative reassurance without operational value
- duplicates nearby labels, states, or metrics
- is a remnant of old hallway-era onboarding language

The following are not valid reasons to remove text:
- “the screen feels busy”
- “modern apps are minimal”
- “we need fewer words”

Text must earn removal by redundancy, not by volume alone.

## 9. Suggested Surface Text Budgets

These are guardrails, not absolute law, but Phase 7 should bias toward them.

### 9.1 Command Surface Budget

- 1 primary headline or section thesis
- 1 concise rationale sentence per major zone
- 0-3 bullets, tags, or reasons per selected object
- no visible paragraph longer than 2 sentences in the core command area

### 9.2 Sheet Budget

- 1 why-now explanation
- 1 compact continuity statement
- 3-5 key evidence/gap rows
- 1 room-entry reason
- 1 dominant CTA

### 9.3 Room First Fold Budget

- 1 title
- 1 subtitle
- bridge + pinned context
- 1 composed setup zone
- no more than 1 deeper explanatory cluster visible before the user starts working

### 9.4 Methodology Budget

- unconstrained by first-fold rules
- but disclosed progressively and intentionally

## 12. Nav / Shell Re-architecture Scope Rules

### 12.1 Allowed Shell Changes

Phase 7 may:
- replace the current left rail as the primary organizing logic
- regroup rooms by family or context
- reduce always-visible destinations
- hide or collapse lower-priority destinations
- strengthen command-first re-entry
- separate utility from room access
- introduce contextual access paths
- move some room links out of the constant shell and into summoned surfaces

### 12.2 Forbidden Shell Changes

Phase 7 may not:
- remove room reachability without a safe alternative access path
- rewrite room internals under the banner of “nav work”
- delete methodology because it does not fit the shell
- flatten module families into vague generic categories
- turn command-first into a thin dashboard over weakened rooms

### 12.3 Required Behavioral Outcome

When Phase 7 is complete, the shell should make the user feel:
- the app knows where I should begin
- the app remembers what I am working on
- the rooms are available when I need them
- I do not have to make 16 equal-weight navigation decisions
- the system is steering me through pressure and continuity, not menu browsing

## 13. High-Risk Copy Categories For Audit

The following categories should be scrutinized aggressively before Phase 7 execution:

### 13.1 Repeated Shell Explanation

Examples:
- repeated “what this product is” copy across command surfaces
- repeated posture/recovery/Week One side explanation blocks
- repeated command-thesis text that users only need occasionally

Likely action:
- compress
- deduplicate
- move some explanation into sheet or help

### 13.2 Room Intro Paragraphs

Examples:
- top-of-room explanation blocks that restate the room thesis every visit
- methodology copy visible before action starts

Likely action:
- compress first-fold text
- move doctrine deeper
- gate some guidance

### 13.3 Footer / Rail / Utility Text

Examples:
- backup, trust, sync, or recovery copy repeated across surfaces
- room-rail or footer explanations that compete with command

Likely action:
- compress
- relocate to utility surfaces
- make less visible until relevant

### 13.4 Duplicated Strategy Language

Examples:
- the same strategic point visible in command, sheet, and room intro
- the same explanation repeated in a hero and a side rail

Likely action:
- pick one canonical layer
- remove duplicates elsewhere

### 13.5 Legacy Hallway-Era Language

Examples:
- copy that exists to orient users inside a module list world
- room labels or explanations that assume the sidebar is the main wayfinding system

Likely action:
- remove or rewrite to command-first language

## 15. Acceptance Gate

Phase 7 may not begin unless the following statements are true:

- we can point to the governing source for every major shell change
- every major text block has been assigned a disclosure decision
- every deep room family has explicit protected substance listed
- connective tissue has an explicit no-break matrix
- no planned cut relies on “modern apps use less text” as its rationale
- command-first behavior is being strengthened without reducing room seriousness
- the shell is changing without the methodology being diluted

If any of the above is false, Phase 7 is not ready.

## 16. Operational Summary

The job of Phase 7 is not to make Antaeus lighter by thinning its thinking.

The job of Phase 7 is to:
- reduce friction
- reduce copy burden
- reduce hallway behavior
- reduce decision clutter

while simultaneously protecting:
- the GTM frameworks
- the module substance
- the methodology
- the strategic seriousness
- the product thesis
- the compounding rules that make the app intelligent

The shell may get cleaner.  
The access model may get smarter.  
The copy may get leaner.  

But the brain and soul of the product must come through more clearly after Phase 7 than before it.
