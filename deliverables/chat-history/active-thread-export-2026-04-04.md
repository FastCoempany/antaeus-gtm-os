## Major user directives, preserved

### On the broader redesign

The user made clear that this was never meant to stop at structural improvement.
The visible design language of the entire product is meant to be rebuilt.

Key user concern:

> i was expecting essentially everything about the ui/ux of the app to change

The user explicitly rejected the idea that the visible state was already final simply because the architecture had improved.

### On the left rail / hallway problem

The user repeatedly pointed out that a hallway of modules was one of the original problems.
The current room rail / family buttons still feel too close to the old module list behavior.

### On the UI being too text-heavy

This line from the assistant was called out by the user as exactly right:

> Too much of the UI still tries to explain itself.

That became a major thread pivot.

### On white space and visual silence

The user pushed for:

- much more white space
- fewer words
- more non-verbal operating-system feel
- less board / essay / Notion-pad energy

### On the visual moodboard direction

The user approved the moodboard direction once the carousel and mark language improved.
Important correction from the user:

- the moodboard was a moodboard
- not a literal page comp
- not an excuse to spread carousel mechanics everywhere

### On the dashboard mode switch

The user specified the intended direction for `Brief / Spotlight / Queue`:

- separate moments
- on a carousel
- tilted forward
- it sort of spins as you click
- all words on each card remain readable

The user also explicitly corrected later misuse of that idea:

- carousel only for this specific part of the app
- not everywhere
- not giant hero cards
- not a page-level composition system

### On semiotics and symbol language

The user wanted stronger non-verbal language and asked:

- what is the UI/UX term for a library of symbols and signs and specific identities for specific components

The working answer in-thread became:

- symbol lexicon
- semiotic system
- design system

### On preserving the soul of the app

The user repeatedly insisted that no redesign should flatten:

- GTM frameworks
- module substance
- methodology
- strategic seriousness
- product thesis
- room brains

This remained a non-negotiable throughout the thread.

## Major assistant positions that were corrected by the user

### 1. Treating structural completion as visible completion

The assistant acknowledged that this was a mistake.
The architecture had improved, but the visible face had not fully caught up.

### 2. Overusing planning / systems language

The user pushed back on language like:

- activation surface
- above the fold
- rank pressure

because it sounded too utilitarian and not enough like visual renovation.

The assistant then reframed the work as:

- facial architecture
- visual composition families
- canonical taste-test surfaces

### 3. Over-explaining in the grammar boards

The user repeatedly rejected:

- manifesto-like board headlines
- rule chips
- too much prose
- explanatory shelves pretending not to be prose

This led to the pivot into quieter, more specimen-led boards.

### 4. Wrong use of moodboard signals

The assistant at one point started carrying the carousel language too literally into a dashboard rebuild.
The user corrected this sharply:

- approved moodboard language is not approved page layout
- the carousel is a local control, not the hero
